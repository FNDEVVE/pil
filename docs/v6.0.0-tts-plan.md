# pi-listen v6.0.0 — TTS support (local + Deepgram)

## Goal
Add text-to-speech to pi-listen so the agent can speak its responses, with the
same backend duality the STT side already has: **local** (sherpa-onnx, offline,
zero-config models) **or Deepgram cloud** (same `DEEPGRAM_API_KEY` users have for
STT, streaming Aura voices). Default to small models that cover popular languages
well.

## Architecture (parallel to existing STT layer)

```
extensions/voice/
  sherpa-engine.ts        ← existing (STT)
  sherpa-loader.ts  NEW   ← extract initSherpa() so STT and TTS share the
                            single-flight module load + the platform checks
                            (ARM32 / musl exclusion). One promise cache covers
                            both engines, no double-init under any race.
  local.ts                ← existing STT model catalog
  tts-local-models.ts NEW ← TTS model catalog (Kitten / Piper-by-language / Kokoro)
  tts-engine.ts     NEW   ← initTts(), getOrCreateTts(modelId), synthesize(),
                            backend dispatch (local vs deepgram)
  tts-deepgram.ts   NEW   ← Deepgram /v1/speak REST + /v1/speak WebSocket clients
                            sharing the same DEEPGRAM_API_KEY resolution as STT
  tts-playback.ts   NEW   ← temp WAV via sherpa.writeWave(), spawn afplay/aplay/
                            powershell SoundPlayer with abort signal
  speak.ts          NEW   ← high-level orchestrator: speak(text, opts) →
                            backend dispatch → engine → playback → abort
voice.ts                  ← register /voice-speak, /voice-speak-stop,
                            /voice-speak-toggle, /voice-speak-test
config.ts                 ← extend with tts* fields + migration
settings-panel.ts         ← add Speak tab + General-tab TTS rows
```

**Why share `sherpa-loader.ts`:** if a user enables both local STT and local TTS,
we don't want two competing dynamic imports of `sherpa-onnx-node`. The
single-flight `??=` promise pattern from the v5.1 sherpa-engine refactor moves
into `sherpa-loader.ts`, both engines call `await loadSherpa()`.

## Deepgram cloud TTS

Deepgram's TTS API uses the same `DEEPGRAM_API_KEY` users already have for STT.
Endpoints:
- **REST**: `POST https://api.deepgram.com/v1/speak?model=<voice>&encoding=linear16&sample_rate=24000&container=wav`
  Body: `{ "text": "..." }`. Returns audio bytes in one shot.
- **WebSocket** (streaming): `wss://api.deepgram.com/v1/speak?model=<voice>&encoding=linear16&sample_rate=24000`
  Send text frames, receive audio frames as they're synthesized — sub-200ms TTFB.

Auth: `Authorization: Token <key>` (same pattern as the existing
`buildDeepgramWsUrl` / `resolveDeepgramApiKey`).

**Voice catalog** (we surface 12 in the picker; Deepgram has more):
- English (Aura-1): asteria, luna, stella, athena, hera, orion, arcas, perseus,
  angus, orpheus, helios, zeus
- Aura-2 multilingual voices when stable

**v6.0 ships with**: REST endpoint only (simpler error model). WebSocket streaming
path is implemented behind a feature flag (`tts.deepgramStreaming = true`) for
users who want sub-200ms TTFB.

**Pricing transparency**: README and onboarding hint mention "$200 free credit"
which is the same credit users get for STT. TTS billing is per-character (not
per-minute like STT) — surface the rate inline so users aren't surprised.

## Audio playback strategy

For a v6.0 MVP: **temp WAV via `sherpa.writeWave()` then spawn the platform
player.** No streaming pipe. Adds ~50-100 ms file-write overhead but keeps the
implementation simple and works identically for both backends (local generates a
WAV; Deepgram REST returns a WAV).

| Platform | Player invocation |
|---|---|
| macOS | `spawn("afplay", [path])` |
| Linux | `spawn("paplay", [path])` with `spawn("aplay", [path])` fallback |
| Windows | `spawn("powershell", ["-NoProfile", "-Command", "$p = $env:PI_SPEAK_PATH; (New-Object Media.SoundPlayer $p).PlaySync()"], { env: { ...process.env, PI_SPEAK_PATH: path } })` |

**Argument-array spawn only — no shell string concatenation.** All three
platforms invoke the player via `child_process.spawn` with a `[command, args]`
array; no template literals, no `shell: true`, no string interpolation of
`path` into a command line. This eliminates the entire class of injection
attacks where a temp directory or filename containing shell metacharacters
(`'`, `"`, `;`, `` ` ``, `$`, newline) would otherwise break out of quoting.
The Windows path uses an environment variable (`$env:PI_SPEAK_PATH`) which
PowerShell resolves at runtime — never substituting it into the command
string itself — so paths like `C:\Users\O'Neil\AppData\Local\Temp\foo.wav`
are passed verbatim to `Media.SoundPlayer`.

Additional defenses on the temp file itself:
- Filenames generated as `pi-speak-${randomUUID()}.wav` — no user input in the
  name, so even if a player did interpret the filename it can't contain
  attacker-controlled bytes
- Path is asserted to live under `os.tmpdir()` via `path.relative()` check before
  spawn — defends against TMPDIR being pointed at a sensitive directory
- Files are written with `0600` perms (owner read/write only). Cleanup uses a
  **single ownership token** (`tempFileOwner` per playback) — the `finally`
  block on the playback promise is the *only* code path that unlinks the file.
  The abort handler kills the player process and clears `activeSpeechProcess`,
  but does NOT unlink — the killed-player path falls through to the same
  `finally` and unlinks there. This eliminates the double-delete race where an
  abort + process exit could both attempt `fs.unlinkSync` and the second one
  throws `ENOENT` (Unix) or `EBUSY` (Windows, file still locked) into an
  unhandled rejection. The `finally` itself wraps the unlink in `try { …
  } catch {}` for the residual case where the file was already removed by an
  external process

Each spawn is held in `activeSpeechProcess` so `/voice-speak-stop` and Escape
can `SIGKILL` it cleanly. WAV files land in `os.tmpdir()` and are unlinked
after playback completes or on next abort, whichever fires first.

**Streaming playback** (start playing while still synthesizing) is a v6.1
optimization. For a 30-word agent response on M3 Pro, the local Kokoro int8
takes ~700 ms total — temp-file overhead is negligible against that.

## Configuration schema additions

```ts
// VoiceConfig in config.ts
ttsEnabled?: boolean;            // default false (opt-in)
ttsBackend?: "local" | "deepgram"; // default "local"
ttsLocalModel?: string;          // default "kitten-nano-en-v0_2"
ttsLocalVoiceId?: number;        // sherpa-onnx sid (numeric); default 0
ttsDeepgramVoiceId?: string;     // Deepgram model id (string); default "aura-asteria-en"
ttsSpeed?: number;               // default 1.0; range 0.5-2.0
ttsAutoSpeak?: boolean;          // speak agent responses automatically; default false
ttsLanguage?: string;            // BCP-47 language tag (e.g. "en", "es-ES", "ja-JP"); default = config.language
ttsDeepgramStreaming?: boolean;  // v6.1 feature flag; default false
```

**Backend-specific voice fields are deliberate.** A single `ttsVoice` typed as
`number | string` would silently pass an `"asteria"` string into sherpa-onnx's
numeric `sid` slot when a user switches Deepgram → Local without re-picking
the voice (and the inverse — passing a numeric `0` as a Deepgram model id).
Keeping the user's Local and Deepgram voice picks independently persisted
eliminates that class of footgun and lets backend-toggles in the settings
panel be lossless.

At config load, `migrateConfig` rejects type-mismatched persisted values:
`ttsLocalVoiceId` must be `typeof === "number"` (else falls back to `0`),
and `ttsDeepgramVoiceId` must be `typeof === "string"` (else falls back to
`"aura-asteria-en"`). `tts-engine.ts` reads only the field that matches the
current backend.

**Language is plumbed end-to-end at synthesize time.** `speak(text, opts)`
resolves the effective language as
`opts.language ?? config.ttsLanguage ?? config.language ?? "en"` and passes
it to the backend dispatch. Each backend uses it as follows:
- **Local (sherpa-onnx)**: for Kokoro multilingual the language goes into
  `OfflineTtsKokoroModelConfig.lang`; for Piper/Kitten (single-language
  models) the language is verified to match the model's bundled language and
  rejected with a clear error if not (e.g. selecting `lang="es-ES"` while
  the active local model is `en_US-lessac-medium` returns
  `"Active local TTS model only supports en. Switch model in /voice-settings or change ttsLanguage."`)
- **Deepgram**: the `model` query parameter on `/v1/speak` already encodes
  the language (e.g. `aura-asteria-en` is en, `aura-luna-es` is es). The
  language field is used to *filter the voice picker* to compatible models
  and to validate that `ttsDeepgramVoiceId` matches `ttsLanguage`. On
  mismatch (user manually edited config to pair `aura-asteria-en` with
  `ttsLanguage: "es"`) the engine returns the same clear-error pattern.

The validation lives in `tts-engine.ts:resolveLanguageForBackend()` which both
playback paths call before any network/native work, so the user gets a
synchronous error rather than 30 seconds of mis-language audio.

`saveConfig` already does atomic writes — no mechanical changes needed there.
The settings panel adds rows to General + a new Speak tab as needed.

## New slash commands

| Command | Purpose |
|---|---|
| `/voice-speak <text>` | One-shot synthesize + play (foreground) |
| `/voice-speak-stop` | Cancel in-flight generation + kill player |
| `/voice-speak-toggle` | Toggle `ttsAutoSpeak` |
| `/voice-speak-test` | Synthesize "The quick brown fox..." in current voice |
| `/voice-settings` | Existing — gains a Speak tab |

Auto-speak after each assistant message ships **disabled by default** in v6.0.
Manual `/voice-speak <text>` is the primary entry point. Auto-speak hook ships
in v6.1 once the manual path is rock-solid in the field.

## Settings panel changes

Models tab gains a `[STT | TTS]` segment toggle at the top — same picker
chassis, two catalogs. No new Tab needed (avoid five tabs).

General tab gets four new rows when `ttsEnabled = true`:
- **TTS Backend** — toggle Local / Deepgram. Switching is lossless: each
  backend's voice id is stored in its own field, so coming back to Local
  restores the previously chosen sherpa sid and going to Deepgram restores
  the previously chosen Aura model.
- **TTS Voice** — opens voice sub-picker. The picker reads from the catalog
  matching the *current* backend: numeric sids (with voice-name labels)
  from `ttsLocalModel` for Local, or Deepgram Aura model ids for Deepgram.
  Saves to the backend-specific config field.
- **TTS Speed** — quick-select 0.5x / 1.0x / 1.25x / 1.5x / 2.0x
- **TTS Test** — synthesize "The quick brown fox jumps over the lazy dog"
  in the current voice + backend.

Reuses the existing language sub-picker pattern for voice selection.

## Risks / non-goals for v6.0.0

- **Streaming local playback** (TTFB optimization) — v6.1
- **Auto-speak hook** — v6.1; v6.0 ships manual `/voice-speak <text>` only to keep
  the surface area small and validated
- **CoreML provider for TTS** — same regression as STT (sherpa-onnx #2910); CPU only
- **Voice cloning / Pocket TTS** — out of scope; different use case
- **Long-form text — robust sentence segmentation.** Naive `.`/`!`/`?` splits
  break on common patterns: `Dr. Smith`, `e.g.`, `v2.0`, `U.S.A.`, URLs like
  `https://example.com`, decimal numbers like `3.14`, file paths like
  `src/foo.ts`. The chunker uses **`Intl.Segmenter(locale, { granularity:
  "sentence" })`** (available in Node 18+ / Bun, locale-aware, no abbreviation
  list to maintain) as the primary path. Fallback for locales without
  Segmenter support: word-count window (chunk every ~25 words at the nearest
  whitespace, never mid-token). Each chunk passes `maxNumSentences = 2` to the
  underlying engine and is queued to playback. The chunker is unit-tested
  against `Dr. Smith said e.g. v2.0 isn't U.S.A.-ready. Visit https://x.dev/p.`
  to lock the regression behavior.
- **TTS while recording** — disabled to prevent feedback loop (mic captures speaker
  output). State machine refuses `/voice-speak` during `recording` / `finalizing`
- **Deepgram WebSocket streaming** — v6.0 ships REST only; WebSocket behind
  `ttsDeepgramStreaming` flag

- New peer-dep range stays the same as 5.1, so it's not a peer-breaking change
- BUT: the package's stated purpose ("hold space to record") expands to bidirectional
  voice (record + speak). Major bump signals the new capability and justifies the
  README banner overhaul
- Additionally: package description in package.json changes from STT-only wording
  to "voice in + voice out for Pi CLI" — that's a metadata break worth signaling
